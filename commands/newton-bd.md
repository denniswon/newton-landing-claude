## Newton Protocol Context

### What Newton Protocol Is

Newton Protocol is a **decentralized policy-based authorization infrastructure** built as an EigenLayer AVS (Actively Validated Service). It enables:

- **Policy Evaluation**: Complex authorization rules in Rego (OPA) language
- **Cryptographic Proof**: BLS signature aggregation from distributed validators
- **Economic Security**: Backed by EigenLayer restaked ETH (billions in TVL)
- **Zero-Knowledge Verification**: SP1/RISC0 proofs for challenge resolution
- **Multi-Chain Support**: Cross-chain policy enforcement

### Core Value Propositions

| Stakeholder Type        | Value Proposition                                                                                                      |
|-------------------------|------------------------------------------------------------------------------------------------------------------------|
| **DeFi Protocols**      | Implement sophisticated on-chain authorization (risk limits, governance, compliance) without rebuilding infrastructure |
| **Enterprises**         | Transaction authorization with audit trails, multi-party approval, regulatory compliance                               |
| **L2 Ecosystems**       | Cross-chain policy enforcement, sequencer-level authorization                                                          |
| **Wallet/AA Providers** | Account recovery policies, daily limits, recipient whitelists, risk-based blocking                                     |
| **Oracles/Bridges**     | Policy-based validation before cross-chain operations                                                                  |

### Key Technical Differentiators

- **Decentralized by default** (no centralized gateway)
- **EigenLayer economic security** (restaking-backed)
- **Rego + WASM** for arbitrary authorization logic
- **BLS aggregation** for cryptographic consensus proof
- **Sub-5s latency** with streaming consensus architecture

## Constraints & Guardrails

### Research Standards

- **Cite sources** for all claims about entity's recent activities, priorities, and pain points
- **Verify timeliness** — prioritize information from the last 6-12 months
- **Distinguish fact from speculation** — clearly label inferences
- **Acknowledge gaps** — state when information is limited or unavailable

### Proposal Standards

- **Be specific** — no generic "blockchain solutions" pitches
- **Be realistic** — proposals must be technically feasible given Newton's current capabilities
- **Be honest** — if Newton is not a good fit, say so and explain why
- **Quantify when possible** — TVL impact, user count, transaction volume affected

### Classification Accuracy

- **Customer**: Would directly integrate Newton as a solution to their authorization/policy needs
- **Partner**: Would collaborate to co-develop, distribute, or enhance Newton (tech, data, GTM, distribution)
- **Both**: Has use cases for direct integration AND strategic partnership value

### Out of Scope

- Do not fabricate quotes, partnerships, or announcements
- Do not make promises about Newton roadmap without caveat
- Do not disclose competitive intelligence inappropriately
- Do not recommend outreach to entities with obvious conflicts of interest

## Execution Steps

### Step 1: Entity Research & Context Gathering

**1.1 Basic Profile**

- What does this entity do? (product/service, market position)
- What blockchain(s) do they operate on?
- What is their scale? (TVL, users, transaction volume, funding)
- Who are their key stakeholders? (founders, BD leads, investors)

**1.2 Recent Activity Analysis (Last 6-12 Months)**

Search for: press releases, blog posts, protocol upgrades, governance proposals, funding rounds, partnerships, security incidents, regulatory developments, hiring patterns, social media discussions by team.

**1.3 Pain Point Identification**

Infer pain points from: publicly stated challenges, competitor moves, product gaps, user complaints, technical limitations, regulatory pressures.

### Step 2: Classification Analysis

**2.1 Customer Fit Assessment**

Score 1-5 on each dimension:

| Dimension                     | Question                                              |
|-------------------------------|-------------------------------------------------------|
| **Authorization Need**        | Do they need policy-based transaction approval?       |
| **Decentralization Priority** | Do they value trustless/decentralized infrastructure? |
| **Economic Security Benefit** | Would EigenLayer backing add credible security?       |
| **Multi-Chain Relevance**     | Do they operate cross-chain or plan to?               |
| **Technical Readiness**       | Can they integrate an AVS (Ethereum/L2 native)?       |

**Customer Score**: Sum / 25 -> High (>0.7), Medium (0.4-0.7), Low (<0.4)

**2.2 Partner Fit Assessment**

| Partnership Type             | Potential Value                                     |
|------------------------------|-----------------------------------------------------|
| **Technology Partner**       | Co-build, integrate tech stacks, shared R&D         |
| **Data Partner**             | Provide data feeds Newton policies can consume      |
| **GTM/Distribution Partner** | Co-marketing, customer referrals, bundled offerings |
| **Ecosystem Partner**        | Joint presence in EigenLayer/ETH ecosystem          |
| **Investment Partner**       | Strategic investment, treasury allocation           |

**2.3 Final Classification**

- **Primary Classification**: Customer / Partner / Both
- **Confidence Level**: High / Medium / Low
- **Reasoning**: 2-3 sentences explaining classification

### Step 3: Opportunity Development

**3.1 For Customers: Integration Proposal**

- Use Case Definition, Technical Integration Path, Value Quantification, Competitive Positioning, Implementation Timeline

**3.2 For Partners: Partnership Proposal**

- Partnership Model, Mutual Benefits, Resource Requirements, Success Metrics, Escalation Path

**3.3 Magic Labs Synergy Identification**

- Bundle opportunities, cross-sell potential, joint enterprise solutions, warm intro paths

**3.4 Landing Page Content Alignment**

- Which newton.xyz sections would resonate most with this entity?
- What messaging adjustments would strengthen the pitch?
- Which use case page (institutional DeFi, stablecoins, AI agents) aligns with their needs?

### Step 4: Action Plan Development

**4.1 Prioritized Recommendations** — Rank by Impact, Feasibility, Urgency, Effort

**4.2 Outreach Strategy** — Target Contact, Entry Point, Opening Message, Supporting Materials

**4.3 Talking Points** — Pain Point Acknowledgment, Solution Connection, Credibility Proof, Differentiation, Call to Action
